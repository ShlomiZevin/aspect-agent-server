# Database Schema: zer4u - Retail Business Intelligence System

## Overview
This is a Hebrew-language retail business intelligence database containing sales, inventory, customer, and employee data. The schema supports comprehensive retail analytics including sales performance, inventory management, customer analysis, and operational metrics.

## ⚡ IMPORTANT: Use Materialized Views for Aggregations (FAST!)

**ALWAYS prefer materialized views over direct sales table aggregation. Direct aggregation on 9.5M rows WILL timeout.**

### Sales by Year (`mv_sales_by_year`) - **USE THIS FOR ANNUAL QUERIES**
Pre-aggregated yearly totals. **ALWAYS use this for annual/yearly questions.**

**Columns:**
- `sale_year` (integer) - Year (e.g., 2023, 2024, 2025)
- `transaction_count` (bigint) - Number of transactions
- `total_revenue` (numeric) - Total sales without VAT
- `avg_revenue` (numeric) - Average transaction value
- `total_cost` (numeric) - Total cost without VAT

**When to use:** "total sales for 2025", "sales this year", "year over year comparison", "annual revenue", etc.

**Example queries:**
```sql
-- Total sales for 2025
SELECT sale_year, total_revenue, transaction_count
FROM zer4u.mv_sales_by_year
WHERE sale_year = 2025;

-- Year over year comparison
SELECT sale_year, total_revenue
FROM zer4u.mv_sales_by_year
ORDER BY sale_year;
```

### Sales by Month (`mv_sales_by_month`) - **USE THIS FOR MONTHLY/PERIOD QUERIES**
Pre-aggregated monthly totals. **ALWAYS use this for monthly/quarterly/period questions.**

**Columns:**
- `year_month` (text) - Period in 'YYYY-MM' format (e.g., '2025-03')
- `sale_year` (integer) - Year
- `sale_month` (integer) - Month number (1-12)
- `transaction_count` (bigint) - Number of transactions
- `customer_count` (bigint) - Number of distinct customers who purchased
- `total_revenue` (numeric) - Total sales without VAT
- `avg_revenue` (numeric) - Average transaction value

**When to use:** "sales this month", "monthly trend", "last 3 months", "sales in January", "monthly comparison", "how many customers this month", etc.

**Example queries:**
```sql
-- Sales and customers for current month
SELECT year_month, total_revenue, transaction_count, customer_count
FROM zer4u.mv_sales_by_month
WHERE year_month = TO_CHAR(CURRENT_DATE, 'YYYY-MM');

-- Last 6 months trend
SELECT year_month, total_revenue, customer_count
FROM zer4u.mv_sales_by_month
WHERE year_month >= TO_CHAR(CURRENT_DATE - INTERVAL '6 months', 'YYYY-MM')
ORDER BY year_month;

-- All months in 2025
SELECT year_month, total_revenue, customer_count
FROM zer4u.mv_sales_by_month
WHERE sale_year = 2025
ORDER BY year_month;
```

### Sales by Store per Month (`mv_sales_by_store_month`) - **USE THIS FOR STORE+TIME QUERIES**
Pre-aggregated store performance by month.

**Columns:**
- `store_number` (integer) - Store ID
- `store_name` (text) - Store name in Hebrew
- `year_month` (text) - Period in 'YYYY-MM' format
- `sale_year` (integer) - Year
- `sale_month` (integer) - Month number
- `transaction_count` (bigint) - Number of transactions
- `total_revenue` (numeric) - Total sales without VAT

**When to use:** "store performance this year", "which store had best sales in January", "monthly breakdown by store", etc.

**Example queries:**
```sql
-- Store performance in 2025
SELECT store_number, store_name, SUM(total_revenue) AS revenue_2025
FROM zer4u.mv_sales_by_store_month
WHERE sale_year = 2025
GROUP BY store_number, store_name
ORDER BY revenue_2025 DESC;
```

### Sales by Store (`mv_sales_by_store`) - **USE THIS FOR ALL-TIME STORE QUERIES**
Pre-aggregated all-time sales totals by store.

**Columns:**
- `store_number` (integer) - Store ID
- `store_name` (text) - Store name in Hebrew
- `transaction_count` (bigint) - Number of transactions
- `total_revenue` (numeric) - Total sales without VAT
- `avg_revenue` (numeric) - Average transaction value

**When to use:** Questions about "top stores", "store ranking", "best stores overall" (no time filter), etc.

**Example queries:**
```sql
-- Top 5 stores by all-time revenue
SELECT store_number, store_name, total_revenue
FROM zer4u.mv_sales_by_store
WHERE total_revenue IS NOT NULL
ORDER BY total_revenue DESC
LIMIT 5;
```

### Sales by Customer (`mv_sales_by_customer`) - **USE THIS FOR CUSTOMER QUERIES**
Pre-aggregated purchases by customer. **ALWAYS use this for customer analysis.**

**Columns:**
- `customer_number` (integer) - Customer ID
- `customer_name` (text) - Customer name
- `purchase_count` (bigint) - Number of purchases
- `total_purchases` (numeric) - Total spending without VAT

**When to use:** Questions about "top customers", "customer spending", "best customers", etc.

**Example queries:**
```sql
-- Top 10 customers by spending
SELECT customer_number, customer_name, total_purchases
FROM zer4u.mv_sales_by_customer
WHERE total_purchases IS NOT NULL
ORDER BY total_purchases DESC
LIMIT 10;
```

### Sales by Product (`mv_sales_by_product`) - **USE THIS FOR ALL-TIME PRODUCT QUERIES**
Pre-aggregated **all-time** sales by product. **No year/date column** — totals span the entire history.

**⚠️ IMPORTANT: mv_sales_by_product has NO date/year column. It cannot be filtered by year or period.**
For year-specific or period-filtered product queries, query `zer4u.sales` + `zer4u.items` directly.

**Columns:**
- `item_code` (text) - Product code
- `item_name` (text) - Product name in Hebrew
- `total_quantity` (numeric) - Total units sold (all time)
- `total_revenue` (numeric) - Total revenue from this product (all time)

**When to use:** "top selling products overall", "best sellers all time", "popular items" (no time filter).
**Do NOT use when question includes:** "this year", "last month", "in 2025", or any time period.

**Example queries:**
```sql
-- Top 10 products all-time by revenue (no time filter)
SELECT item_code, item_name, total_revenue, total_quantity
FROM zer4u.mv_sales_by_product
ORDER BY total_revenue DESC
LIMIT 10;

-- Top products in a specific year — must use sales + items directly
SELECT s.item_code, i.item_name, SUM(s.revenue) AS total_revenue, SUM(s.quantity) AS total_quantity
FROM zer4u.sales s
JOIN zer4u.items i ON s.item_code = i.item_code
WHERE s.sale_date >= '2025-01-01' AND s.sale_date < '2026-01-01'
  AND s.item_code IS NOT NULL
GROUP BY s.item_code, i.item_name
ORDER BY total_revenue DESC
LIMIT 10;

-- Top products this year
SELECT s.item_code, i.item_name, SUM(s.revenue) AS total_revenue
FROM zer4u.sales s
JOIN zer4u.items i ON s.item_code = i.item_code
WHERE s.sale_date >= DATE_TRUNC('year', CURRENT_DATE)
  AND s.item_code IS NOT NULL
GROUP BY s.item_code, i.item_name
ORDER BY total_revenue DESC
LIMIT 10;
```

### Sales by Day (`mv_sales_by_day`) - **USE THIS FOR DAILY QUERIES**
Pre-aggregated daily sales for the last 90 days. **ALWAYS use this for daily trend questions.**

**Columns:**
- `sale_date` (date) - Calendar date
- `transaction_count` (bigint) - Number of transactions
- `total_revenue` (numeric) - Total revenue (without VAT)
- `avg_revenue` (numeric) - Average transaction value

**When to use:** "daily sales", "sales last 7 days", "day by day breakdown", "daily trend", etc.

**Example queries:**
```sql
-- Daily sales for the last 7 days
SELECT sale_date, transaction_count, total_revenue
FROM zer4u.mv_sales_by_day
WHERE sale_date >= CURRENT_DATE - INTERVAL '7 days'
ORDER BY sale_date DESC;
```

## Core Business Tables

### Sales Data (`sales` - 69M records)
**⚠️ WARNING: Direct aggregation on this table is VERY SLOW. Use materialized views above for aggregated queries!**

Primary transaction table. **Key columns have been migrated to English with proper types — use them directly:**

- `sale_date` (DATE) — fully indexed, use for all date filters: `WHERE s.sale_date >= '2025-01-01'`
- `store_id` (INTEGER) — join with stores: `ON s.store_id = st.store_id`
- `customer_id` (INTEGER) — join with customers: `ON s.customer_id = c.customer_id`
- `item_code` (TEXT) — join with items: `ON s.item_code = i.item_code`
- `revenue` (NUMERIC) — revenue without VAT, use directly: `SUM(s.revenue)`
- `cost` (NUMERIC) — cost without VAT
- `quantity` (NUMERIC) — units sold

**⚠️ NEVER use old Hebrew column names for these fields** — `תאריך מקורי SALES`, `מס.חנות SALES`, `מס.לקוח`, `קוד פריט SALES`, `מכירה ללא מעמ` are the old names that cause errors. The migrated English columns above are what you MUST use.

### Items/Products (`items` - 28K records)
Product catalog and pricing:
- **Identity**: `item_code` (item code - TEXT), `item_name` (item name in **Hebrew** - TEXT)
- **Classification**: `item_group` (item group/category in Hebrew - TEXT), `תת קבוצת פריט` (sub-group)
- **Pricing**: Multiple price fields (`מחיר עלות 0`, `מחיר עלות 6`, `מחיר מחירון 1/2/4`)
- **Suppliers**: `שם ספק` (supplier name)
- **⚠️ Item names are in Hebrew** — when user searches in English (e.g. "milk"), note that the catalog is in Hebrew and suggest the Hebrew equivalent (e.g., חלב). Still run the query but explain the result.

**Example queries:**
```sql
-- Top categories by revenue
SELECT i.item_group, SUM(s.revenue) AS total_revenue, COUNT(*) AS transactions
FROM zer4u.sales s
JOIN zer4u.items i ON s.item_code = i.item_code
WHERE i.item_group IS NOT NULL
GROUP BY i.item_group
ORDER BY total_revenue DESC NULLS LAST
LIMIT 10;

-- Search by item name (Hebrew)
SELECT item_code, item_name, item_group
FROM zer4u.items
WHERE item_name ILIKE '%ורד%'
LIMIT 20;
```

### Inventory (`inventory` - 19.8M records)
Stock levels and values per warehouse entry:
- **Keys**: `InventoryKey` (links to sales.InventoryKey and min_inventory.InventoryKey), `InventoryMahsanKey`
- **Metrics**: `יתרת מלאי` (stock balance - TEXT, cast to numeric), `ערך מלאי` (stock value)
- **⚠️ No item code column** — use `mv_inventory_by_item` for all inventory+item queries (much faster!)

### Inventory by Item (`mv_inventory_by_item`) - **USE THIS FOR ALL INVENTORY QUERIES**
Pre-aggregated stock per item. **ALWAYS use this instead of querying inventory directly.**

**Columns:**
- `item_code` (text) - Item code
- `item_name` (text) - Item name in Hebrew
- `total_stock` (numeric) - Current stock balance (negative = out of stock)
- `total_value` (numeric) - Total stock value
- `min_stock` (numeric) - Minimum required stock (NULL if not defined)

**Example queries:**
```sql
-- Items with low or negative stock
SELECT item_code, item_name, total_stock, min_stock
FROM zer4u.mv_inventory_by_item
WHERE total_stock <= 0
ORDER BY total_stock
LIMIT 50;

-- Items below minimum stock
SELECT item_code, item_name, total_stock, min_stock
FROM zer4u.mv_inventory_by_item
WHERE min_stock IS NOT NULL AND total_stock < min_stock
ORDER BY (total_stock - min_stock)
LIMIT 50;

-- Total inventory value
SELECT SUM(total_value) AS total_inventory_value,
       COUNT(*) AS distinct_items,
       SUM(CASE WHEN total_stock <= 0 THEN 1 ELSE 0 END) AS out_of_stock_items
FROM zer4u.mv_inventory_by_item;
```

### Minimum Inventory (`min_inventory`)
Minimum stock thresholds per inventory key:
- **Keys**: `InventoryKey`
- **Threshold**: `MLI_MINIMOM` (minimum required stock - TEXT, cast to numeric)
- **Usage**: Already included in `mv_inventory_by_item.min_stock` — no need to join directly

### Customers (`customers` - 1.4M records)
Customer information. **English columns to use:**
- `customer_id` (INTEGER) — primary key, join with sales: `ON s.customer_id = c.customer_id`
- `customer_name` (TEXT) — customer name

**Hebrew columns (must be double-quoted in SQL):**
- `"ישוב"` — city/settlement (use for geographic analysis)
- `"כתובת"` — address
- `"מין"` — gender

**JOIN example:**
```sql
SELECT c.customer_name, c."ישוב" AS city, SUM(s.revenue) AS total_revenue
FROM zer4u.sales s
JOIN zer4u.customers c ON s.customer_id = c.customer_id
WHERE s.sale_date >= CURRENT_DATE - INTERVAL '12 months'
GROUP BY c.customer_name, c."ישוב"
ORDER BY total_revenue DESC
LIMIT 20;
```

### Stores (`stores` - 94 records)
Store master data. **English columns to use:**
- `store_id` (INTEGER) — primary key, join with sales: `ON s.store_id = st.store_id`
- `store_name` (TEXT) — store name in Hebrew

**Hebrew columns (must be double-quoted in SQL):**
- `"תאריך סגירת חנות"` — closing date (DD/MM/YYYY text)
- Other columns: area manager, store type, opening date — all Hebrew, rarely needed

**JOIN example:**
```sql
SELECT st.store_name, SUM(s.revenue) AS total_revenue
FROM zer4u.sales s
JOIN zer4u.stores st ON s.store_id = st.store_id
GROUP BY st.store_name
ORDER BY total_revenue DESC;
```

**For top N products per store (use ROW_NUMBER):**
```sql
WITH ranked AS (
  SELECT s.store_id, st.store_name, s.item_code, i.item_name,
         SUM(s.quantity) AS total_qty,
         ROW_NUMBER() OVER (PARTITION BY s.store_id ORDER BY SUM(s.quantity) DESC) AS rnk
  FROM zer4u.sales s
  JOIN zer4u.stores st ON s.store_id = st.store_id
  JOIN zer4u.items i ON s.item_code = i.item_code
  WHERE s.sale_date >= CURRENT_DATE - INTERVAL '12 months'
  GROUP BY s.store_id, st.store_name, s.item_code, i.item_name
)
SELECT store_id, store_name, item_code, item_name, total_qty
FROM ranked WHERE rnk <= 3
ORDER BY store_id, rnk;
```

**⚠️ No "active" column** — active stores have closing date >= today: `WHERE zer4u.parse_date_ddmmyyyy("תאריך סגירת חנות") >= CURRENT_DATE`

## Supporting Tables

Date dimension with Hebrew date fields:
- **Base**: `תאריך מקורי` (original date), `DateN` (date number)
- **Periods**: `שנה` (year), `חודש` (month), `רבעון` (quarter), `שבוע` (week)

- **`units_delivery_costed`**: Delivery unit costs and employee assignments

### Financial Tables
- **`hesbonithiuvi`**: Invoice accounting records
- **`multips`**: Multi-purchase discounts
- **`shorot_kbla`**: Credit card payment details

## Key Relationships
After migration, the main sales columns are typed — use them directly, no cast needed:
- Sales ↔ Items: `s.item_code = i.item_code` (both TEXT)
- Sales ↔ Customers: `s.customer_id = c.customer_id` (both INTEGER — direct join, no cast!)
- Sales ↔ Stores: `s.store_id = st.store_id` (both INTEGER — direct join, no cast!)
- Sales ↔ Inventory: via `s."InventoryKey"` (text composite key)
- Sales ↔ Targets: via `s."TargetKey"` (text)

## Important Notes
1. **Migrated columns in sales** (use directly, no cast, fully indexed):
   - `store_id` (integer), `customer_id` (integer), `item_code` (text)
   - `sale_date` (date) — use direct date comparison: `s.sale_date >= '2025-01-01'`
   - `revenue` (numeric), `quantity` (numeric), `cost` (numeric)
2. **Hebrew columns** still exist in sales (many unmigrated fields) — avoid them for main queries
3. **Item names are in Hebrew** — `item_name` in items table is Hebrew text
4. **Nullability**: All columns allow NULL values
5. **Performance**:
   - ✅ USE materialized views for aggregations (instant results)
   - ✅ USE `sale_date` for date filters (indexed DATE column)
   - ✅ USE `store_id`, `customer_id` directly for joins (indexed integers)
   - ❌ AVOID direct aggregation on full sales table without date filter (slow on 69M rows)

## Query Strategy
1. **For annual totals** ("sales for 2025", "this year"): Use `mv_sales_by_year` — instant!
2. **For monthly/period queries** ("this month", "last 3 months", "monthly trend", "how many customers this month"): Use `mv_sales_by_month` (has `customer_count` column) — instant!
3. **For store+time queries** ("store performance in 2025"): Use `mv_sales_by_store_month` — instant!
4. **For all-time aggregations by store/customer/product**: Use `mv_sales_by_store` / `mv_sales_by_customer` / `mv_sales_by_product`
   **⚠️ For year/period-filtered product queries** ("top products this year"): `mv_sales_by_product` has NO date column — use `sales JOIN items` with `sale_date` filter directly
5. **For daily trends** ("daily sales", "sales by day", "last 7 days"): Use `mv_sales_by_day` — instant!
6. **For inventory queries** ("stock levels", "items below minimum", "out of stock"): Use `mv_inventory_by_item` — instant!
7. **For active stores count**: `SELECT COUNT(*) FROM zer4u.stores WHERE store_id > 0` (or use a closing date filter if needed)
8. **For categories by revenue**: JOIN sales + items on item_code, GROUP BY item_group
9. **For returning vs new customers**: Use MIN(sale_date) per customer — see example below
10. **NEVER aggregate directly on sales table without a narrow filter** — use the materialized views!

## Common Analysis Patterns
```sql
-- Total sales for 2025
SELECT total_revenue FROM zer4u.mv_sales_by_year WHERE sale_year = 2025;

-- Sales this month
SELECT total_revenue FROM zer4u.mv_sales_by_month WHERE year_month = TO_CHAR(CURRENT_DATE, 'YYYY-MM');

-- Top stores by revenue
SELECT * FROM zer4u.mv_sales_by_store ORDER BY total_revenue DESC LIMIT 10;

-- Top customers
SELECT * FROM zer4u.mv_sales_by_customer ORDER BY total_purchases DESC LIMIT 10;

-- Best selling products
SELECT * FROM zer4u.mv_sales_by_product ORDER BY total_quantity DESC LIMIT 10;

-- Categories by revenue (item_group)
SELECT i.item_group, SUM(s.revenue) AS total_revenue, COUNT(*) AS transactions
FROM zer4u.sales s
JOIN zer4u.items i ON s.item_code = i.item_code
WHERE i.item_group IS NOT NULL
GROUP BY i.item_group
ORDER BY total_revenue DESC NULLS LAST
LIMIT 15;

-- Stores that sold the most popular items last month
SELECT st.store_name, i.item_name, SUM(s.quantity) AS qty_sold
FROM zer4u.sales s
JOIN zer4u.stores st ON s.store_id = st.store_id
JOIN zer4u.items i ON s.item_code = i.item_code
WHERE s.sale_date >= DATE_TRUNC('month', CURRENT_DATE - INTERVAL '1 month')
  AND s.sale_date < DATE_TRUNC('month', CURRENT_DATE)
  AND s.item_code IN (
    SELECT item_code FROM zer4u.mv_sales_by_product ORDER BY total_quantity DESC LIMIT 5
  )
GROUP BY st.store_name, i.item_name
ORDER BY qty_sold DESC
LIMIT 20;

-- Returning vs new customers for last month — FAST pattern (avoids full table scan)
-- Step 1: find customers who purchased last month (small set, uses index)
-- Step 2: find their first-ever purchase (scoped to that small set only)
WITH month_customers AS (
  SELECT DISTINCT customer_id
  FROM zer4u.sales
  WHERE sale_date >= DATE_TRUNC('month', CURRENT_DATE - INTERVAL '1 month')
    AND sale_date < DATE_TRUNC('month', CURRENT_DATE)
    AND customer_id IS NOT NULL
),
first_purchases AS (
  SELECT s.customer_id, MIN(s.sale_date) AS first_purchase
  FROM zer4u.sales s
  INNER JOIN month_customers mc ON s.customer_id = mc.customer_id
  GROUP BY s.customer_id
)
SELECT
  COUNT(CASE WHEN first_purchase >= DATE_TRUNC('month', CURRENT_DATE - INTERVAL '1 month') THEN 1 END) AS new_customers,
  COUNT(CASE WHEN first_purchase < DATE_TRUNC('month', CURRENT_DATE - INTERVAL '1 month') THEN 1 END) AS returning_customers
FROM first_purchases;
-- ⚠️ NEVER use the pattern: GROUP BY customer_id on the full sales table + EXISTS — it causes a timeout on 69M rows

-- Items below minimum stock
SELECT item_code, item_name, total_stock, min_stock,
       (total_stock - min_stock) AS shortage
FROM zer4u.mv_inventory_by_item
WHERE min_stock IS NOT NULL AND total_stock < min_stock
ORDER BY (total_stock - min_stock) ASC
LIMIT 50;
```
